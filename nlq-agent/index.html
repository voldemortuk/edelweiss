Day 1 · Frame

Prompt Engineering
for text-to-SQL

Turn a finance question into correct SQL with nothing but a well-engineered prompt — and learn exactly why it breaks.
Day 1 · Prompt Engineering
The prompt is the program
Engineering-relevant test cases
  • Schema-grounded SQL: "top 5 overdue vendors by amount" → correct JOINs & filters
  • Ambiguity handling: "last quarter" → resolve fiscal calendar, not literal
  • Refusal & safety: reject "delete all GL rows" / out-of-scope asks
  • Structured output: enforce a Pydantic/JSON contract, every time
Build #1 — end of Day 1
A prompt-only text-to-SQL function with a validation layer & a failure-mode test suite.
Day 2 · Ground

RAG & Vector DB —
the precursor to agents

A prompt can't hold the whole warehouse. Retrieve the right schema, glossary & examples just-in-time.
Day 2 · RAG & Vector DB
Retrieve the context, then generate
Why this comes before agents
  • Grounding: retrieve only the tables a question needs — no 200-table prompt
  • Chunking & indexing: schema, glossary & golden queries, hybrid (dense+sparse, RRF)
  • Eval harness: Precision@K / Recall@K + groundedness on every answer
  • Hallucination drop: measure the lift retrieval gives vs Day 1 baseline
Build #2 — end of Day 2
A retrieval layer + eval harness that measurably beats the prompt-only baseline.
Day 3 · Orchestrate

Agents —
plan, generate, validate

One LLM call isn't enough for a number you'll defend. Split the job across role-based agents that check each other.
Day 3 · Multi-Agent
A team of agents, with a feedback loop
Orchestration concepts
  • Role-based agents: planner · executor · critic, each tuned to its job
  • LangGraph state: shared context across nodes/edges + the retry loop
  • MCP tool layer: structured, access-controlled tool & DB calls — and A2A messaging
  • Validator-in-the-loop: regenerate on failure — correctness before answers
  • Single vs multi-agent: when the extra hop is worth it (and when it isn't)
Build #3 — end of Day 3
A LangGraph multi-agent NLQ pipeline (Planner → SQL-Gen → Validator) with a self-correcting loop.
Day 4 · Ship

Production —
adapt, control & deploy

Choose the right model strategy (prompt vs RAG vs fine-tune), keep a human in the loop, then wrap it in guardrails, observability & cost control finance can trust.
Day 4 · Production
From a notebook to a service finance trusts
Adapt, control & ship
  • Model strategy: prompt vs RAG vs fine-tune (LoRA/QLoRA/PEFT) decision framework
  • Human-in-the-loop: approval flows, interrupt handling; conversational & voice (STT/TTS) front-ends
  • Guardrails: PII detection, prompt- & SQL-injection defence, output validation
  • Observability: LangSmith tracing; token/latency/failure logging; A/B
  • Security: OAuth/Managed Identity, row/column security, audit (SOX/GDPR), Key Vault
  • Cost: token tracking, semantic caching, model routing (small↔large)
Capstone — end of Day 4
A deployed, guardrailed, observable NLQ service — demoed live on Edelweiss finance data.
